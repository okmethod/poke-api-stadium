# 物理エンジン（2D）の設計

## レイヤー別の役割分担

```
ユースケース
  ↓ DI（ファクトリ関数で注入）
Port インターフェース: 物理エンジンに非依存の境界
  ↑ 実装
Adapter
```

ユースケースは Port インターフェースにのみ依存するため、**エンジンを差し替えてもアプリ層・ドメイン層・プレゼン層への変更は不要**。
具象アダプター2ファイルを書き直し、ファクトリ関数を差し替えるだけで移行可能。

---

## Matter.js の現状課題

- ポリゴン衝突の不安定性: `fromVertices` による凸分解が複雑な輪郭で失敗しやすい。現状は `COLLISION_SCALE = 0.6` で輪郭を 60% に縮小して回避している
- 高速ボディの壁抜け（トンネリング）: CCD（連続衝突検出）非対応のため、高速移動するボディが薄い壁をすり抜ける場合がある
- 拘束の数値不安定性: `Constraint` のエネルギーが蓄積すると振動・発散が起きることがある（シーソー拘束に影響）
- メンテナンス停滞: Matter.js の開発は事実上停滞しており、バグ修正・機能追加が期待しにくい

---

## Rapier（WASM）への移行検討

### メリット

- ポリゴン精度の向上: 凸分解アルゴリズムが堅牢。`COLLISION_SCALE` による縮小ハックを撤廃できる
- CCD 対応: ボディ単位で連続衝突検出を有効化でき、高速ボディの壁抜けを防止できる |
- 拘束の安定性: `RevoluteJoint` がシーソーに適しており、数値的に安定している |
- 決定論的シミュレーション: 同一入力で必ず同一結果。リプレイや結果検証ロジックに応用できる
- センサーコライダー: 物理応答なしの重なり検出ゾーンを定義できる
- 活発なメンテナンス: Rust 製コアで開発が継続されており、機能追加・バグ修正が期待できる

### デメリット・移行コスト

- アセットサイズ増加: WASM バイナリ約 600KB〜1MB が追加される（Matter.js は約 80KB）
- API の再実装: Matter.js と Rapier では API パラダイムが異なる（RigidBody + Collider モデル）
- 衝突イベントの書き直し: Matter.js の `Events.on` 相当がなく、Rapier の `EventQueue` 方式で再実装が必要
- ドラッグ操作の自前実装: `MouseConstraint` 相当が存在せず、自前でドラッグ拘束を実装する必要がある
- シーソー拘束の書き直し: `RevoluteJoint` で対応可能だが API が異なり、実装の書き直しが必要

### 移行スコープ

アーキテクチャ上、変更が必要なのは**インフラ層のみ**。

- 変更不要: ドメイン層・アプリ層（Port・ユースケース）・プレゼン層
- 新規作成: アダプター
- 差し替え: 各ユースケースへの DI
